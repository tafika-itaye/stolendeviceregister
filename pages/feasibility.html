1. Executive Summary

DeviceWatch is a national stolen device registry — a public-facing web platform and mobile application allowing citizens to register stolen IT devices (phones, laptops, tablets) by IMEI or serial number, verified against national identity and police records. Buyers can instantly check any second-hand device before purchase.

The concept is technically highly feasible with your stated skill set (C#, HTML, JavaScript, SQL). The core technology is a verified database with a public search API — similar in scope to haveibeenpwned.com but with identity verification.

Market gap: No single authoritative stolen device registry exists publicly in most African markets. This creates immediate value for second-hand markets, law enforcement, and insurers.

2. Technical Feasibility

2.1 Core System Requirements

The system has three distinct layers:

Frontend (Client)

  • HTML5 / CSS3 / Vanilla JS
  • Progressive Web App (PWA)
  • Mobile-responsive design
  • Offline IMEI cache

Backend API

  • ASP.NET Core (C# ✓)
  • REST API with JWT auth
  • Background jobs (Hangfire)
  • Image upload (Azure Blob)

Database

  • SQL Server / Azure SQL (✓)
  • Indexed IMEI/Serial columns
  • Audit log table
  • Encrypted PII columns

Integrations

  • National ID verify API
  • SMS OTP (Africa's Talking)
  • File/image CDN
  • Email (SendGrid)

3. Core Database Schema (SQL)

Key tables required (simplified):

-- Users table (PII encrypted at rest)
Users (Id, Email, PasswordHash, NationalId_Encrypted,
       PhoneNumber, FullName, IsVerified, CreatedAt)

-- Devices registry (core table - heavily indexed)
StolenDevices (Id, UserId FK, DeviceType, MakeModel,
               IMEI [INDEXED], SerialNumber [INDEXED],
               PoliceReportNumber, TheftDate,
               IdentifyingMarks, PhotoUrl,
               Status [Active/Resolved/Disputed],
               RegisteredAt, LastUpdated)

-- Search audit log (for analytics + abuse detection)
SearchLog (Id, SearchTerm, SearchType [IMEI/Serial],
           IPAddress, HitFound, Timestamp)

-- Verification log
VerificationLog (Id, UserId, NIDVerified, ReportVerified,
                 VerifiedAt, APIResponse)

4. External API Integrations

4.1 National ID Verification

Note: Kenya's IPRS (Integrated Population Registration System) API requires formal application and MOU with the government. For South Africa use Home Affairs API. In the demo/MVP phase, implement syntax validation + manual verification workflow as a fallback.

Estimated integration cost: KES 0 (government partnership) — apply through relevant ICT Authority or request API access from the national registry body. Budget 3–6 months for approval.

4.2 Police Report Validation

Currently no public API exists. Recommended approach:

Phase 1: Validate by syntax/format only (regex: region/station/year/number). Phase 2: Partner with police service for API access or manual verification by a police liaison. Phase 3: Full API integration.

4.3 SMS OTP — Africa's Talking

Cost: ~KES 0.8/SMS. For 1,000 registrations/month = ~KES 800/month. API is well-documented for .NET.

# → Deploy to Azure App Service from GitHub Actions

# → CI/CD pipeline auto-deploys on push to main
